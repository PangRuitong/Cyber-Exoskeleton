## 清单

### D-1 并发：两个我同时跑会怎样？
- **抓什么**：竞态条件。任何 check-then-act 模式（先查后插、先读后改）在两个实例同时执行时都有时间缝。
- **初级为什么漏**：单人单线程的心智模型；本地测试永远只有一个请求。
- **教义**：并发正确性只能靠**站在必经之路上的原子机制**（数据库唯一索引、事务、锁），never 靠应用层逻辑。
- **案例**：SPEC-08 FR-7——应用层查重存在竞态缝隙，双击 slash 命令 = 两实例双双漏过；唯一索引 + ON CONFLICT 在数据库内原子裁决。

### D-2 崩溃/重启：每两步之间被 kill 会怎样？
- **抓什么**：非原子多步流程的中间态。逐段分析：步骤 A 完成、步骤 B 未开始，此刻进程死了，世界处于什么状态？谁来收拾？
- **初级为什么漏**：默认代码从头跑到尾；"部署"和"平台重启"不在心智模型里。
- **教义**：为每个 kill 窗口指认一个恢复机制（重试 + 幂等 / backfill / 人工可见）。找不到恢复机制的窗口 = 设计缺陷。
- **案例**：SPEC-08——"插入成功、回执未达"窗口由 FR-7 幂等接住；"原始码已发、翻译未发"窗口由 FR-8 静默放弃语义接住（用户手里已有保底）。

### D-3 网络：超时是未知态，不是失败态
- **抓什么**：把"没收到响应"当"操作失败"的错误归类。超时时对端可能已成功。
- **初级为什么漏**：本地网络从不抖，超时分支从没被真正走过。
- **教义**：未知态一律按失败**报告**（触发用户/系统重试），但 never 按失败**假设**（如据此补偿性回滚）；重试的安全性由幂等保障——D-3 依赖 D-4。
- **案例**：SPEC-08 FR-3——插入调用超时按失败报告，用户重发由 content_hash 幂等无害化。

### D-4 重复执行：这段逻辑跑两遍，世界会多出什么？
- **抓什么**：重试来源（人、平台、cron、部署重启）× 副作用类型。逐个副作用问：执行两次和一次结果是否相同？
- **初级为什么漏**：把"正常只会执行一次"当承诺；不区分只读副作用与状态变更。
- **教义**：先写"重复执行语义备忘"（谁会重试、窗口在哪），再设计幂等键。幂等键的**作用域要匹配重复窗口的作用域**（见 D-6）。迁移脚本一律幂等（IF NOT EXISTS），重跑无害。
- **案例**：SPEC-08——Discord 平台不重试，重复唯一来源是人，窗口 = kill 窗口 → 日期桶 content_hash；INV-5 论证报告类副作用（发消息）天然无需幂等。

### D-5 攻击者：每个输入的信任边界画在哪？
- **抓什么**：不只用户输入——**错误消息、日志、缓存键**都可能载有不可信内容。攻击者视角逐个入口问：我能塞什么进来、能让系统替我花什么（钱、调用、磁盘）？
- **初级为什么漏**：只在"登录/权限"语境想安全；想不到防护机制本身（如去重缓存）也有攻击面。
- **教义**：处理顺序即安全：**先剥离/规范化不可信内容，再做任何以它为键的操作**（hash、缓存、日志）。资源消耗类防线要分层（去重是第一防线，预算闸门是最后防线）。
- **案例**：SPEC-08 FR-6——错误回显值先剥离再 hash；顺序反了 = 攻击者用畸形载荷制造无限不同 hash 击穿去重缓存，逐条烧 LLM 调用。

### D-6 作用域匹配：解法的作用域等于问题的作用域吗？
- **抓什么**：用永久机制解临时问题、用全局机制解局部问题（或反之）。作用域错配的解法今天能跑，副作用日后才显形。
- **初级为什么漏**：解法"能解决眼前问题"就收工，不问它顺带禁止了什么。
- **教义**：写下问题的作用域（时间/空间/范围），解法作用域超出的部分逐一问"这个超出禁止了什么合法行为？"
- **案例**：SPEC-08 INV-3——kill 窗口是分钟级，永久全局去重会静默吞噬数月后的合法重捕获 → 日期桶。

### D-7 静默 vs 喊叫：最坏的故障是不喊的那种
- **抓什么**：静默数据丢失、静默降级、静默吞异常。崩溃会喊，喊了就能修；静默故障你永远不知道。
- **初级为什么漏**：把"不报错"当"正确"；catch 块吞掉异常让程序"稳定"。
- **教义**：每个"操作被跳过/合并/降级"的分支必须留下可见痕迹（响应字段、回执标注、日志行）。评审时专门找"这里失败了但没人知道"的路径。
- **案例**：SPEC-08——dedup 回执显式标注（吞掉不说 = 用户以为存了新想法）；迁移 never 静默失败或静默删数据；FR-8 静默放弃也必须留日志一行。

### D-8 依赖审计：我改的这个东西，还有谁在读它？
- **抓什么**：改变一个共享物（列、函数契约、环境变量、算法语义）时的隐性读者。语义改变比结构改变更阴——结构变了会编译报错，语义变了照常运行、悄悄错。
- **初级为什么漏**：只看自己要改的调用点；grep 了名字但没逐处判定语义。
- **教义**：改共享物前全仓库 grep，**逐处判定新语义下是否仍正确**并留痕。"引用了但没审"等于没 grep。
- **案例**：SPEC-08 阶段 2——content_hash 换算法（纯内容 → 掺日期），enrich-thought 按旧语义查重的代码若不审计会静默失效。同类：SPEC-06 PK 迁移 × call-llm upsert 的共同部署陷阱。

### D-9 契约兼容：改造前先把现状钉成字节级基线
- **抓什么**："向后兼容"没有凭据就是口号。改造后"看起来一样"不等于一样。
- **初级为什么漏**：凭印象记现状；改完只测新路径不回测旧路径。
- **教义**：动任何被调用的东西之前，先原样记录现有调用方**实际传什么、实际收什么**（字段、默认值、来源优先级），作为验收时的比对凭据。现状不清楚时先侦察再动手（只读侦察单先行）。
- **案例**：SPEC-08——AC-6 基线钉死为 enrich-thought/weekly-digest 的实际调用样式；侦察单 A 发现 effort 已有 env 机制、content_hash 是生成列，两处都推翻了 spec 假设，避免了实施期返工。
